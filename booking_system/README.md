## День 3 — Логика программы и пользовательский сценарий

### Версия 7: Главный цикл и управление логикой программы

#### 1. Логика программы

На данном этапе была выстроена полная логика взаимодействия с пользователем.

Основой стал цикл `while True`, который управляет всем сценарием работы программы:

- программа не завершается после одного действия  
- пользователь может повторять выбор  
- управление остаётся внутри одного потока выполнения  

В процессе работы я:

- поняла роль `while True` как главного управляющего цикла  
- разобралась, где правильно завершать сценарий с помощью `break`  
- выстроила последовательность действий внутри цикла  
- реализовала повтор сценария при отказе пользователя  

**Логика работы:**  
`вывод → ввод → проверка → подтверждение → действие`

**Цель:**

- реализовать непрерывный пользовательский сценарий  
- сделать поведение программы предсказуемым  
- сохранить контроль над выполнением на каждом этапе  

---

#### 2. Пример кода

```python
all_times = ["10:00", "11:00", "12:00", "13:00"]
busy_times = ["11:00"]
free_times = []

# собираем свободные слоты
for time in all_times:
    if time not in busy_times:
        free_times.append(time)

while True:

    # вывод слотов
    for index in range(len(free_times)):
        number = index + 1
        print(f"{number} - {free_times[index]}")

    # ввод
    selected_number = input("Выберите удобное время (введите номер): ").strip()

    # проверка: число
    while not selected_number.isdigit():
        print("Ошибка: введите цифру.")
        selected_number = input("Выберите удобное время (введите номер): ").strip()

    # проверка: диапазон
    while not 0 < int(selected_number) <= len(free_times):
        print("Ошибка: такого номера нет.")
        selected_number = input("Выберите удобное время (введите номер): ").strip()

    # получаем время
    index = int(selected_number) - 1
    selected_time = free_times[index]

    # подтверждение
    confirmation = input(f"Подтвердите запись на {selected_time} (да/нет): ").strip().lower()

    if confirmation == "да":
        busy_times.append(selected_time)
        print("Вы записаны на:", selected_time)
        break
    else:
        print("Выбор отменён, попробуйте снова.")
```

## День 4 — Работа со списками и оптимизация кода

### Версия 8: Оптимизация через list comprehension и обработка крайних случаев

### Знакомство с list comprehension

На этом этапе я продолжила практику на Stepik, решала задачи на работу со списками и условиями.

С инструментом `list comprehension` я столкнулась не напрямую в заданиях, а случайно — увидела решение другого ученика, которое выглядело как одна строка вместо привычного цикла `for`.

Это вызвало интерес, и я начала разбираться:

- как работает такая запись  
- что она делает  
- чем отличается от обычного цикла  

В процессе анализа пришла к пониманию, что:

`list comprehension` — это способ создавать списки более кратко и читаемо по сравнению с классическим циклом.

### Пример: классический подход

```python
numbers = [1, 2, 3, 4, 5, 6]

filtered_numbers = []

for number in numbers:
    if number % 2 == 0:
        filtered_numbers.append(number ** 2)

print(filtered_numbers)
```

---

### Тот же код через list comprehension

```python
numbers = [1, 2, 3, 4, 5, 6]

filtered_numbers = [number ** 2 for number in numbers if number % 2 == 0]

print(filtered_numbers)
```

---

### Вывод

- `list comprehension` сокращает код
- делает его более компактным
- позволяет объединить фильтрацию и преобразование в одной строке

При этом важно учитывать читаемость и не усложнять выражения.

## День 5 — Set, оптимизация и улучшение логики ввода

### Версия 9: Использование set для оптимизации и улучшение валидации ввода

### Неожиданное знакомство с set

На этом этапе я продолжила практику на Stepik и столкнулась с новым для себя инструментом — `set` (множество).

Как и в случае с `list comprehension`, это произошло не напрямую через задание, а случайно: я увидела решение, где использовались фигурные скобки, и это вызвало вопросы.

Я начала разбираться:
- чем `set` отличается от `list`
- как он работает
- в каких ситуациях его использовать

---

### Что я поняла

`set` — это структура данных, которая:
- хранит только уникальные значения
- не сохраняет порядок элементов
- работает быстрее при проверке `"есть элемент или нет"`

Главное отличие от списка:
- `list` → перебирает элементы по порядку
- `set` → выполняет быструю проверку (без перебора)

---

###  Дополнительное открытие: разница между set и dict

В процессе работы я столкнулась с неожиданным моментом: фигурные скобки в Python используются не только для `set`, но и для `dict`.

Ранее я считала, что `{}` — это всегда множество, но оказалось:
- `{}` → это пустой словарь (`dict`)
- `set()` → это пустое множество (`set`)
- `{"кот", "заяц"}` → это множество (`set`)
- `{"кот": 1}` → это словарь (`dict`)

Также на практике проявилось свойство автоматического удаления дубликатов:

```python
a = {"кот", "кот", "заяц"}
print(a)
# Результат: {'кот', 'заяц'}
```
Это помогло мне лучше понять различие структур данных и избежать ошибок при их инициализации.

---

### Применение в задачах

Пример фильтрации с использованием `set` внутри `list comprehension`:
```python
animals = ['слон', 'заяц', 'волк', 'кот']
allowed_animals = {'заяц', 'кот'}

filtered_animals = [animal for animal in animals if animal in allowed_animals]
print(filtered_animals)
```

**Важное понимание:**
- `list comprehension` → создаёт список (результат)
- `set` → используется внутри условия для быстрой проверки
- Итог: `set` ускоряет проверку, `list` хранит итоговую выборку

## Исправление логики ввода (ключевое улучшение)

Ранее использовались два отдельных цикла `while`:

```python
while not selected_number.isdigit():
    ...
while not 0 < int(selected_number) <= len(free_times):
    ...
```

### Проблема
После повторного ввода пользователь мог ввести нечисловое значение, и программа падала с ошибкой при попытке выполнить `int(selected_number)` во втором цикле.

### Решение
Оба цикла объединены в один устойчивый блок:

```python
while True:
    selected_number = input("Выберите удобное время (введите номер): ").strip()

    if not selected_number.isdigit():
        print("Ошибка: введите цифру.")
        continue

    if not 0 < int(selected_number) <= len(free_times):
        print("Ошибка: такого номера нет в списке.")
        continue

    break  # Ввод корректен, выходим из цикла валидации
```

### Почему это важно
- каждая новая попытка ввода проверяется с нуля
- полностью исключена ошибка `ValueError`
- логика становится предсказуемой и устойчивой
- код становится чище и понятнее

# День 7 | Booking System V11 | Начало работы с сохранением данных

В предыдущей версии была поставлена задача сохранять данные между запусками программы. Сегодня началась реализация этой функциональности.

До текущего момента все записи существовали только в оперативной памяти. После завершения программы содержимое `busy_times` полностью терялось. Поэтому было принято решение реализовать сохранение занятых слотов в файл.

## Архитектурные решения

Перед написанием кода была продумана логика будущей системы хранения данных:

- Во время работы программы источником правды (Source of Truth) является `busy_times`.
- После завершения программы актуальное состояние сохраняется в файле `busy_times.txt`, из которого данные будут восстановлены при следующем запуске.
- Файл хранит текущее состояние системы, а не историю изменений.
- При изменении данных файл будет полностью перезаписываться.
- Планируется проверять существование файла.
- Если файл отсутствует, программа должна создать его автоматически.
- Если файл существует, данные будут загружаться из него в `busy_times`.

Отдельно было принято решение использовать полную перезапись файла, а не добавление новых данных в конец.

Такой подход позволяет всегда сохранять актуальное состояние системы без накопления устаревших данных.

Например, если слот `11:00` уже содержится в `busy_times`, то при очередном сохранении программа должна записать актуальное состояние множества занятых слотов целиком.

Поэтому для проекта был выбран режим записи `"w"` (write), который полностью обновляет содержимое файла.

Использование режима добавления (`append`, `"a"`) привело бы к накоплению старых данных и усложнило бы восстановление текущего состояния системы.

Также был подробно разобран будущий поток данных:

Загрузка данных:

Файл → str → split("\n") → list → set → busy_times

Сохранение данных:

busy_times → join("\n") → str → файл


Но очень быстро стало понятно, что такой вариант подходит не всегда.

Если файла не существует, Python завершит выполнение программы с ошибкой.

Получается, перед чтением данных сначала необходимо убедиться, что файл вообще существует.

Но теперь проект сам подвёл к новой теме.

Пришлось разобраться, какие ещё инструменты предоставляет Python для работы с файлами и операционной системой.

Во время этого обсуждения неожиданно выяснилось, что с модулями Python я уже сталкивалась раньше.

Например, в учебных задачах уже использовалась конструкция:

```python

```

Я понимала, что оператор `import` подключает библиотеку с готовыми инструментами.

Но только сейчас стало понятно, что `random` — это такой же модуль Python, как и `os`.

Разница лишь в том, что каждый модуль решает свои задачи.

`random` помогает работать со случайными числами.

`os` предоставляет инструменты для взаимодействия с операционной системой и файловой системой.

Именно поэтому в проекте впервые появились строки:

```python

```

и

```python
os.path.exists("busy_times.txt")
```

Функция `os.path.exists()` позволила разделить запуск программы на два независимых сценария.

Если файл уже существует — программа загружает сохранённые данные.

Если файла ещё нет — автоматически создаёт новый и только после этого продолжает работу.

Стало понятно, что проект сам начинает подсказывать, какие темы необходимо изучать дальше.

# 🏗 Архитектурное решение

Во время обсуждения появился важный вопрос.

> **Что является настоящим источником данных: память программы или файл?**

Сначала казалось, что главным объектом является `busy_times`.

Однако постепенно стало понятно, что множество существует только во время работы программы.

После завершения процесса оно полностью исчезает.

Файл же продолжает существовать независимо от программы.

Именно поэтому было принято архитектурное решение.

> **Файл `busy_times.txt` становится Source of Truth всей системы.**

Это означало ещё одно важное изменение в логике программы.

Недостаточно изменить множество `busy_times`.

После каждого изменения состояние системы должно немедленно синхронизироваться с постоянным хранилищем.

Во время обсуждения появилась ещё одна мысль.

Если программа неожиданно завершится, произойдёт ошибка или компьютер выключится, изменения, которые остались только в памяти, будут потеряны.

Поэтому было принято решение не откладывать сохранение до конца работы программы.

Каждое подтверждение записи должно сразу сохраняться в файл.

# 🧩 Почему понадобился join()

Во время реализации нового механизма возникла ещё одна задача.

Файл умеет хранить только текст.

Однако после подтверждения записи данные находятся в множестве `busy_times`.

Возник вопрос.

> **Как превратить множество обратно в обычный текст?**

Именно для этого понадобился метод `join()`.

Он объединяет несколько строк в одну большую строку, вставляя между элементами выбранный разделитель.

В проекте в качестве разделителя используется символ новой строки.

```python
busy_times_text = "\n".join(busy_times)
```

После этого вся информация снова становится обычным текстом и может быть сохранена в файл.
